## B) Root Issue

`AstTypeAnnotation::element_kind` (`ast_type.hpp:74`) was declared as a field on every AST node but was **never populated in any write site**. All compound type annotations (Array, List, Map, Set, Vector, Stack, Queue, Tuple, Pointer) carried `element_kind::Unknown`.

This meant:
- Array literals `[1, 2, 3]` → `kind=Array, element_kind=Unknown`
- Collection constructors `list(1, 2)` → `kind=List, element_kind=Unknown`
- Function returns of compound types → no element type metadata
- Variable references to arrays → no element type propagation
- For-loop range expressions → no element type on range node

The TypeChecker already determined element types internally during inference (first element of array literals, first arg of collection constructors) but discarded this information before writing the annotation.

---

## C) Architecture Rationale

### Design Decisions

1. **`AstTypeKind` parameter (not `AuroraType`)**: The `element_kind` field stores `AstTypeKind`, not `AuroraType`. Using `AstTypeKind` directly avoids a round-trip conversion. Both `annotate_node()` and `annotate_ret()` accept `AstTypeKind element_kind_val = AstTypeKind::Unknown` with a default of Unknown, preserving full backwards compatibility with all 58+ existing call sites.

2. **Per-variable element type tracking**: A `var_element_types_` map (`std::unordered_map<std::string, AstTypeKind>`) was added to TypeChecker alongside the existing scopes system. This enables:
   - Storing element kind at assignment time
   - Retrieving element kind at variable reference time
   - Propagation through memory operations (Delete/Move/etc.)
   
   Two new methods were added: `define_var_elem()` and `lookup_var_elem()`.

3. **Annotation-first propagation**: Element kind flows through the annotation on AST nodes. When a compound type is created (array literal, collection constructor), element_kind is set on the node. When that node is used in an assignment, the element_kind is read from the RHS node's annotation and propagated to the LHS node and to the variable element type map.

4. **Heterogeneous -> Unknown**: When a tuple or array has elements of different types, element_kind is set to Unknown. The TypeChecker never guesses.

## K) Final Verdict

**H2 Phase E-1 is COMPLETE.**

Element kind metadata now flows through the entire TypeChecker annotation pipeline for every compound type node:

- **Literals**: Array and tuple literals carry element_kind from the first element (with homogeneity checking for tuples)
- **Constructors**: `array()`, `tuple()`, `list()`, `set()`, `vector()`, `stack()`, `queue()` propagate element kind from the first argument
- **Builtins**: `split()`, `fields()`, `methods()` return `Array<String>`; `unique()`, `filter()` propagate from input; all other compound-return builtins propagate from first arg
- **Assignments**: Element kind propagates from RHS annotation to LHS annotation and to per-variable element type storage
- **Variable references**: Element kind is retrieved from storage and set on the reference node
- **For-loop**: Element kind propagates from range expression annotation
- **Memory operations**: Delete/Move/Drop/Borrow/Copy/Free/Reference/Pointer — element kind propagates from the looked-up variable or expression
- **Returns/Output/Throw**: Element kind propagates from the value expression

The `annotate_node()` and `annotate_ret()` functions accept an optional `AstTypeKind element_kind_val` parameter defaulting to `Unknown`, preserving full backwards compatibility.

**Build**: 0 errors, 0 new warnings

**Behavioural change**: NONE (element_kind has zero codegen consumers in the current codebase)

3. **No codegen changes needed yet** — element_kind is ready for consumption when Phase E-2 or E-3 introduces consumers
