| Gap | Impact |

## 7. Remaining Debt

### Honest assessment of remaining gaps:

1. **`gen_function()` LLVM signature** ‚ÄĒ still hardcodes `i8ptr_ty()` return and `i64_ty()` params. The annotation on the Function node is available but not consumed by gen_function's LLVM type selection. Changing this would affect Aurora's internal calling convention and needs a careful ABI-compatible phase.

2. **`StructLiteral` field types** ‚ÄĒ resolved from string names in `codegen_struct_gep()` (union field access). The `StructInfo` carries `type_name` strings per field, and field-level annotations (`element_kind`) could carry the per-field type. But that requires extending the annotation schema beyond the current single `AstTypeAnnotation` per node.

3. **`optimized_codegen.cpp`** ‚ÄĒ zero annotation awareness. All types hardcoded to `i64`. This is a separate backend that needs its own `get_annotation_kind()`-style migration (H2 Phase C follow-up).

4. **`ast_to_ir.cpp`** ‚ÄĒ hardcoded `i64` for everything. Separate IR system with no annotation awareness.

5. **Class vtable entries** ‚ÄĒ method type signatures in vtables are hardcoded to `i64` return and `i64` params. Annotations on declared method types could eventually drive vtable signatures.

6. **Generic/template declarations** ‚ÄĒ no annotation support for generic type parameters or template instantiations. Requires full generic type system.

7. **Field-level `AstTypeAnnotation`** ‚ÄĒ each AST node carries one annotation. Struct field declarations (child nodes of StructDecl) don't have independent annotations that per-field codegen can read. An `element_kind` abuse could work for arrays but not general structs.

8. **`Import` / `NamespaceDecl` / `ModuleDecl` / `AliasDecl`** ‚ÄĒ not annotated. These are structural/compile-time nodes with no runtime type information.

## 10. Next Recommendation

### H2 Phase E ‚ÄĒ Richer Type Metadata

**Justification:** The annotation pipeline is now complete for expression and declaration nodes. The next bottleneck is the quality and richness of the metadata itself:

1. **Element type propagation** ‚ÄĒ `AstTypeAnnotation::element_kind` is set to `Unknown` in all `annotate_node` calls. Arrays, lists, maps, sets, vectors, stacks, queues all carry `element_kind::Unknown`. Populating this requires tracking element types through array literals, generics, and function returns.

2. **Field-level annotations** ‚ÄĒ Struct fields, class fields, and union members need per-field `AstTypeKind`. The current single-annotation-per-node design doesn't support this. A schema extension (e.g., `std::vector<AstTypeAnnotation> field_annotations` or an `AnnotationMap`) would enable codegen_types.cpp and codegen_oop.cpp to prefer per-field annotations over string/boolean dispatch.

3. **`gen_function()` LLVM type selection** ‚ÄĒ With richer element-type and return-type metadata, `gen_function()` can begin selecting LLVM types based on annotations rather than hardcoding `i8ptr_ty()`/`i64_ty()`.

4. **Generic/template type tracking** ‚ÄĒ Annotations for generic type parameters (`T`, `U`) and their instantiations. This is foundational for Aurora's generic type system but requires design work beyond simple metadata.

Phase E should focus on (1) and (2) as the highest-impact, lowest-risk changes. (3) and (4) require more careful design and testing.
