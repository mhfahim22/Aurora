## B) Root Issue

Three field metadata structures existed in the codebase, but none carried structured `AstTypeKind` metadata in a complete way:

| Structure | File | `type_kind`? | `element_kind`? |
|-----------|------|-------------|-----------------|
| `StructFieldInfo` | `type_registry.hpp:8-15` | ❌ Missing | ❌ Missing |
| `ClassFieldInfo` | `class_oop.hpp:44-52` | ✅ (H2 D2) | ❌ Missing |
| `UnionFieldInfo` | *does not exist* | — | — |

`StructFieldInfo` still relied entirely on:
- `is_string` / `is_float` boolean flags (legacy approach)
- `type_name` string-based type annotation (e.g., `"int"`, `"float"`, `"Point"`)

`ClassFieldInfo` had `type_kind` (added in H2 D2) but was missing `element_kind` for compound field types.

Two codegen consumers were still reading `fi->is_string` directly (bypassing the `type_kind`-aware helpers):

| File | Line | Legacy usage |
|------|------|-------------|
| `codegen_expr.cpp` | 271 | `return fi ? fi->is_string : true;` |
| `codegen_function.cpp` | 180 | `m.returns_string = fi ? fi->is_string : true;` |

## C) Architecture Rationale

### Design Decisions

1. **Extend existing structs, no new parallel maps**: Both `StructFieldInfo` and `ClassFieldInfo` received new fields directly. No `var_element_types_`-style parallel map was introduced. This follows the rule: "prefer extending existing metadata structures."

2. **Prefer type_kind in codegen, fall back to legacy**: The `struct_field_llvm_type()` helper prefers `f.type_kind` when known, falls back to `f.type_name` string for extern structs, then to `i64` default. This mirrors the existing `field_llvm_type()` pattern in `codegen_oop.cpp`.

3. **Raw AST node type inference for StructDecl fields**: Since `register_struct()` is called from `walk_stmt` before the struct body's expression nodes are visited by `infer_expr`, the `type_annotation` on RHS nodes is not yet populated. Type inference is therefore done from the raw node type (Str → String, Float → Float, Num → Int), consistent with the existing `is_string`/`is_float` pattern.

4. **Extern type_name to AstTypeKind mapping**: Added `extern_type_name_to_kind()` in `typechecker_types.cpp` that maps FFI type name strings to `AstTypeKind`. Handles all types recognized by `extern_type_to_llvm()` in `codegen_types.cpp`.

5. **Union codegen type lookup**: The inline type mapping in `codegen_struct_gep()` for union field bitcasts was updated to prefer `f.type_kind` before falling back to string-based type name lookup.

## K) Final Verdict

**H2 Phase E-2 is COMPLETE.**

Every declaration field now carries structured type metadata:

| Metadata | `StructFieldInfo` | `ClassFieldInfo` |
|----------|------------------|-----------------|
| `type_kind` (resolved `AstTypeKind`) | ✅ ADDED (H2 E-2) | ✅ PRE-EXISTING (H2 D2) |
| `element_kind` (for compound fields) | ✅ ADDED (H2 E-2) | ✅ ADDED (H2 E-2) |

**Population:**

- **Extern struct/union fields**: `type_kind` resolved from FFI type name string via `extern_type_name_to_kind()` (supports Int, Float, String, Bool, Void). `element_kind` remains Unknown (extern types are scalar/opaque).
- **StructDecl fields**: `type_kind` inferred from raw node type (Str → String, Float → Float, Num → Int). `element_kind` remains Unknown (all defaults are scalars). A new explicit `else if (Num)` branch was added (previously Num was in the catch-all `else`).
- **Class fields**: `type_kind` (already set in H2 D2) preserved; `element_kind` added but stays Unknown for all current class field defaults (all scalars).

**Consumer wiring (annotation-first preference):**

- `get_or_create_struct_type()`: reworked field type resolution to use `struct_field_llvm_type()` — prefers `f.type_kind`, falls back to `f.type_name` string, then `i64`.
- `codegen_struct_gep()` (union path): prefers `f.type_kind` over string-based type name lookup.
- `expr_is_string_type()` (Attribute path): prefers `fi->type_kind` over `fi->is_string`.
- Method return type inference in `codegen_function.cpp`: prefers `fi->type_kind` over `fi->is_string`.

All codegen paths maintain complete backward compatibility through fallback chains (type_kind → legacy flags → defaults).

**Build**: 0 errors, 0 new warnings  
**Tests**: 17/17 PASS (100%)  
**Behavioural change**: NONE — additive metadata only; zero ABI/IR/parser impact

### Future Cleanup Recommendations

1. **Remove legacy `is_string`/`is_float` booleans** from `StructFieldInfo` and `ClassFieldInfo` once `type_kind` coverage reaches 100%. This is a low-risk mechanical change that can be done after H2 E-3.
2. **Merge `StructFieldInfo.type_kind` resolution with the annotation pipeline** — once the TypeChecker's analysis pass runs before `register_struct()`, field types can be populated from annotation instead of raw node types.
3. **Wire `element_kind` consumption** when compound field types (e.g., `Array<Int>` fields) are supported in the parser/type system.
